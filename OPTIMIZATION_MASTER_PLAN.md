# Pi Gateway 永不停歇优化主计划

## 核心哲学
- **永远执行**：除非达到"世界最好代码"，否则永不停止
- **简单至上**：用最简单代码实现，拒绝复杂机制
- **严格重构**：5次以上重构，不增删功能只优化结构
- **完备测试**：每个修改都有可观测的测试结果
- **超时保障**：所有操作必须有超时，防止挂起
- **定期压缩**：上下文定期压缩，保持高效执行

## 第一阶段：完备测试验证（当前会话）

### 1.1 Session 匹配机制测试
```
测试目标：验证严格session匹配机制正确工作
测试用例：
1. 单客户端单session - 消息正常接收
2. 单客户端切换session - 旧session消息缓冲，新session消息接收
3. 多客户端不同session - 消息不串发
4. 客户端重连 - 缓冲消息正确flush
5. 并发消息 - 各session消息独立

可观测指标：
- 控制台日志：[PiAgentSession.send] Client has NOT selected session / Client selected session
- 消息到达率：100%匹配session的消息到达
- 消息串发率：0%
```

### 1.2 内存泄漏测试
```
测试目标：验证无内存泄漏
测试用例：
1. 长时间运行(1小时) - 内存稳定
2. 频繁切换session(100次) - 内存无增长
3. 多连接并发(10个) - 内存正常

可观测指标：
- 进程内存使用：< 500MB
- 无重复定时器
- 无累积的Map条目
```

### 1.3 性能测试
```
测试目标：验证性能达标
测试用例：
1. 消息发送延迟 < 50ms
2. session切换 < 2s
3. 列表加载 < 500ms

可观测指标：
- 响应时间日志
- 无阻塞操作
```

## 第二阶段：代码重构计划（5轮）

### 第1轮：命名与结构规范
目标：统一命名，清晰结构
- 统一session相关命名（shortId, sessionId, sessionFile）
- 统一status命名（runtimeStatus, connectionStatus）
- 函数命名动词开头
- 文件职责单一化

### 第2轮：消除重复代码
目标：DRY原则，提取公共逻辑
- 提取共同的错误处理
- 提取共同的验证逻辑
- 提取共同的转换逻辑
- 合并相似的handler

### 第3轮：简化复杂度
目标：降低认知负担
- 减少嵌套层级（最大3层）
- 简化条件判断
- 使用早期返回
- 消除魔法数字

### 第4轮：依赖优化
目标：降低耦合
- 减少跨模块引用
- 明确接口边界
- 依赖注入替代全局状态
- 单向数据流

### 第5轮：极致简化
目标：代码最小化
- 删除所有未使用代码
- 删除所有过度设计
- 删除所有注释掉的代码
- 用最直接的方式实现

## 第三阶段：测试代码完善

### 3.1 单元测试

### 3.2 集成测试

- WebSocket消息流测试

- 性能指标收集
